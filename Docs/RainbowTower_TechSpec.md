# RainbowTower Technical Specification

## 1. Document purpose

This document is the implementation source of truth for the Unity prototype `RainbowTower`.
It is written specifically for staged execution by an AI coding agent such as Codex 5.3.

Primary implementation rule:
- the agent must implement exactly one stage per run;
- each completed stage must leave the project in a runnable state;
- after each stage the agent must stop, report the result, ask for testing, fix reported bugs, and continue only after explicit user confirmation.

This document is intended to be used together with:
- `AGENTS.md`
- `Docs/CodexStagePrompts.md`
- visual references:
  - `Docs/template.png`
  - `Docs/fake_screenshot.png`

## 2. Product overview

- Project name: `RainbowTower`
- Genre: `2D Tower Defense`
- Platform: `Android`
- Orientation: `Portrait`
- Target audience: `casual / mid-core`
- Theme: `fantasy, magic crystals, goblins`
- Engine: `Unity`

Prototype goal:
- validate the core loop of defending a central tower through crystal-driven mana generation and automatic color attacks.

Secondary goals:
- keep the prototype readable and easy to extend;
- minimize risky scene/prefab churn;
- keep feature boundaries clear for AI-assisted iteration.

## 3. High-level game concept

The player defends a central magical tower placed inside a square arena. Enemies spawn from the top portal area, follow a fixed looped path around the tower, and attempt to reach the exit. The tower fires automatically when mana is available. Mana is generated by crystals placed on three shelves in the lower HUD.

Core fantasy:
- top shelf crystals generate base colors;
- middle shelf crystals convert base mana into mixed mana;
- bottom shelf crystal converts all colors into white mana;
- stronger colors unlock later and increase the tactical and economic depth.

## 5. Core gameplay rules

### 5.1 Enemy rules

- Enemies spawn from the entrance portal.
- Enemies move along a fixed predefined path.
- Reaching the exit reduces player HP.
- Enemies appear in waves.
- Each next wave scales:
  - enemy HP;
  - reward XP;
  - optionally count or spawn interval.

### 5.2 Tower rules

- There is exactly one central tower.
- The tower fires automatically.
- Fire cadence: `1 shot every 2 seconds` by default.
- A shot can only happen if at least one valid mana point is available.
- If several colors are available, the tower shoots in a deterministic queue/rotation.
- Each shot consumes `1 mana` of the chosen color.
- Target priority: enemy closest to exit.
- Damage type: direct instant hit.

Attack colors:
- `Red`
- `Green`
- `Blue`
- `Yellow`
- `Magenta`
- `Cyan`
- `White`

### 5.3 Crystal and mana rules

Top shelf:
- `Red`
- `Green`
- `Blue`

Middle shelf:
- `Yellow = Red + Green`
- `Magenta = Red + Blue`
- `Cyan = Green + Blue`

Bottom shelf:
- `White = all colors`

Base generation:
- `1 mana per second` per active crystal before upgrades or modifiers.

Generation rules:
- top shelf crystals generate without input cost;
- middle shelf crystals consume `1 + 1` of their parent base colors to produce `1` mana of their own color;
- white crystal consumes `1` mana of every non-white color to produce `1 White`;
- generation pauses when required inputs are missing;
- each crystal has a mana cap;
- locked crystals do not generate.

### 5.4 Progression rules

XP:
- gained from killed enemies;
- spent on unlocking crystals and upgrading crystals.

Level effects:
- increased mana generation speed;
- increased mana cap;
- increased damage.

Damage formula:
- base crystal shot damage = `damage value from the crystal config for its current level`;
- mixed crystal shot damage = `own current-level damage value from config + current-level damage values from both parent crystal configs`;
- white shot damage = `sum of current-level damage values from config for all crystals participating in the full chain`.

Damage configuration rule:
- crystal damage must not be derived from the numeric level index itself;
- each crystal level must read its gameplay values, including damage, from configuration data for that level;
- generation speed and mana cap should follow the same pattern when practical.

### 5.5 Optional active mana collection

Tap-drops are optional and must be implemented only in the dedicated later stage.

Rules:
- tower shots have a configurable spawn chance of `10% to 25%`;
- drop spawns around the tower;
- drop color is selected from unlocked colors;
- lifetime is `3 seconds`;
- each drop gives `+1 mana` of its color on tap;
- active simultaneous drops are capped;
- mechanic should add efficiency without becoming mandatory.

## 6. Technical architecture

### 6.1 Core principles

- Runtime logic must not depend on Editor-only APIs.
- Runtime asset access must go through serialized references from scene objects, prefabs, or ScriptableObjects provided via `ConfigurationProvider`.
- Dependencies between gameplay controllers should be explicit and injected during construction or initialization.
- `MonoBehaviour` classes should mainly hold scene references, visual bindings, and Unity lifecycle bridges.
- Plain C# controller/model classes should own gameplay rules where practical.

### 6.2 Bootstrap chain

Required bootstrap objects in the main scene:

#### `GameManager` (`MonoBehaviour`)

Responsibilities:
- entry point of runtime bootstrap;
- defines creation and initialization order of all manager/controller classes;
- defines update order for non-MonoBehaviour gameplay systems;
- handles orderly deinitialization.

Requirements:
- added to main scene;
- has serialized reference to `ServiceLocator`;
- registers itself into `ServiceLocator` first.

#### `ServiceLocator` (`MonoBehaviour`)

Responsibilities:
- stores references to shared runtime services and scene providers;
- provides access to scene `MonoBehaviour` dependencies and configuration access;
- is passed into managers/controllers during creation.

Requirements:
- added to main scene;
- has serialized reference to `ConfigurationProvider`;
- registers `ConfigurationProvider` second.

#### `ConfigurationProvider` (`ScriptableObject`)

Responsibilities:
- stores references to feature configuration ScriptableObjects;
- provides centralized access to configuration assets.

Requirements:
- contains typed references to configuration assets created by stages;
- does not load assets dynamically through editor APIs or Resources unless the user explicitly changes the rule later.

### 6.3 UI root

#### `MainUiProvider` (`MonoBehaviour`)

Responsibilities:
- single serialized root of gameplay UI;
- provides direct field access to all UI elements used from code;
- all UI elements must be direct or indirect children under it.

Requirements:
- added to main scene;
- registered in `ServiceLocator`;
- contains:
  - `FloatingTextParent`
  - `HudParent`
  - `PopupParent`
- has `Canvas` in `Screen Space - Overlay`;
- has `CanvasScaler` with reference resolution `1920 x 1080`.

Code rule:
- all code access to gameplay UI must go through `MainUiProvider` fields or components referenced from it.

## 7. Required feature-based folder structure

All implementation must live under `Assets/Project/`.

Each feature must have its own folder:
- `Assets/Project/<FeatureName>/`

Subfolders inside each feature:
- `Assets/Project/<FeatureName>/<FeatureName>Src/`
- `Assets/Project/<FeatureName>/<FeatureName>Pfs/`
- `Assets/Project/<FeatureName>/<FeatureName>Art/`
- `Assets/Project/<FeatureName>/<FeatureName>Cfg/`

Rules:
- ScriptableObject `.asset` files belong only in `*Cfg`;
- C# classes defining ScriptableObjects belong in `*Src`;
- prefabs belong in `*Pfs`;
- sprites, animations, animator controllers, particles and other visual assets belong in `*Art`.

## 9. Main runtime systems

### 9.1 Enemy System

Responsibilities:
- spawn enemy views from configured prefabs;
- move enemies along the fixed path;
- expose current path progress for targeting;
- notify on death or escape.

### 9.2 Wave System

Responsibilities:
- define wave sequence;
- control spawn timing;
- scale HP and rewards;
- notify UI about wave state and completion.

### 9.3 Tower System

Responsibilities:
- select best target;
- request available attack color from mana/crystal systems;
- apply cooldown;
- calculate and apply damage;
- trigger shot feedback and optional drop spawn events.

### 9.4 Mana System

Responsibilities:
- store mana per color;
- enforce caps;
- handle generation and spending;
- expose read-only values to UI.

### 9.5 Crystal System

Responsibilities:
- define crystal unlock state and levels;
- control generation behavior;
- resolve attack rotation order across unlocked colors;
- provide color damage calculation inputs.

### 9.6 Progression System

Responsibilities:
- track XP;
- spend XP on unlocks and upgrades;
- expose progression actions to UI;
- synchronize unlock dependencies.

### 9.7 Combat Feedback

Responsibilities:
- floating text;
- hit effects;
- attack effects;
- damage flash;
- insufficient resource feedback;
- optional sound hooks.

### 9.8 Tap Drops

Responsibilities:
- spawn optional tap collectibles;
- manage lifetime and cap;
- process taps through a dedicated input path;
- grant mana on successful collection.

## 13. Delivery rules for every stage

At the end of every stage, Codex must:

1. stop implementation immediately after the current stage scope is complete;
2. not implement any requirement from later stages;
3. report:
   - implemented scope;
   - added/changed files, prefabs, ScriptableObjects, scenes, folders;
   - current gameplay or interaction loop available to the player;
   - unverified items due to lack of Unity/editor execution;
4. ask the user to:
   - test the stage in Unity or a build;
   - report bugs or UX issues;
   - confirm when to continue.

If the user reports bugs for the current stage:
- fix those bugs before moving to the next stage.
